# 钻石 ERP 实施路线图

## 1. 路线图目标

本路线图用于把当前原型从“可演示页面集合”推进到“可落地业务系统蓝图”。

路线图重点回答：

1. 当前最优先补什么
2. 为什么按这个顺序做
3. 每一阶段的交付物是什么
4. 每一阶段完成的判断标准是什么

## 2. 当前项目判断

当前项目已经完成了以下基础：

- 左侧统一导航结构
- 采购、在途、分选、细筛、封包、出货的页面骨架
- 扫码交接中心与异常单中心
- 交接留痕、异常留痕、企业微信预留结构

当前项目还没有完成以下决定业务成败的部分：

- 采购批次正式创建入口
- 客户订单正式创建入口
- 香港收货与深圳收货的正式关口
- 统一的单据自动生成规则
- 统一的流程状态机和任务待办机制

因此，下一阶段不建议继续零散补页面样式，而应先收束业务起点和流程中枢。

## 3. 实施原则

### 原则 1：先定单据，再做页面

如果单据来源和关系不稳定，页面越做越容易返工。

### 原则 2：先定关口，再做联动

先把采购、收货、分选、细筛、封包、出货的关键关口定清楚，再做自动生成和自动联动。

### 原则 3：先定状态机，再接企业微信

企业微信只是确认方式，不应反过来决定业务结构。

### 原则 4：先让系统可走通，再让系统更自动

先完成“人工可走通的一条主线”，再引入自动派单、自动提醒、自动异常。

## 4. 阶段划分

## 阶段 0：方案冻结

### 目标

冻结订单来源、单据关系、角色边界、状态机和主流程。

### 交付物

- 业务蓝图文档
- 实施路线图文档
- 单据关系与角色责任确认结果

### 完成标准

- 明确系统采用 `采购批次 + 客户订单` 双起点
- 明确旧 `订单管理` 是否改造为 `客户订单管理`
- 明确 `收货管理` 是否新增为一级业务关口

## 阶段 1：重构业务起点

### 目标

把“订单怎么来的”真正落地到系统入口。

### 开发项

1. 新增 `采购批次创建`
2. 新增 `客户订单创建`
3. 改造旧 `订单管理`

### 建议页面

- 采购批次列表
- 新建采购批次
- 客户订单列表
- 新建客户订单

### 为什么先做

因为没有源头单据，后面所有流程单据都只能继续依赖 mock 推导。

### 完成标准

- 操作员可在系统中手工创建采购批次
- 销售或跟单可在系统中手工创建客户订单
- 旧 `订单管理` 不再承担供应链主线入口

## 阶段 2：补收货关口

### 目标

补齐印度到香港、香港到深圳、深圳入库之间最关键的实操关口。

### 开发项

1. 新增 `香港收货`
2. 新增 `深圳收货`
3. 新增 `收货差异复核`
4. 新增 `收货异常登记`

### 为什么先做

收货是供应链从“在途”变成“在库”的分界点，也是后续分选领料是否合法的前置条件。

### 完成标准

- 在途记录到港后可创建收货单
- 收货单可记录实收重量、实收粒数、差异与复核人
- 收货完成后才能进入分选待办

## 阶段 3：建立统一状态机

### 目标

让系统从“页面按钮改状态”升级为“流程规则驱动状态”。

### 开发项

1. 统一采购、在途、收货、分选、细筛、封包、出货状态定义
2. 统一交接状态定义
3. 统一异常冻结和关闭规则
4. 建立状态推进映射表

### 为什么先做

没有统一状态机，后面自动派单、库存联动、企业微信确认都无法稳定落地。

### 完成标准

- 每类单据都有明确状态集合
- 每一次推进都有合法前置条件
- 不允许随意回退，只允许异常冻结

## 阶段 4：建立待办与自动派单

### 目标

把当前“页面推导待交接任务”的方式，升级为“系统统一待办”。

### 开发项

1. 自动生成待收货任务
2. 自动生成待分选任务
3. 自动生成待细筛任务
4. 自动生成待封包任务
5. 自动生成待出货任务
6. 自动生成待扫码交接任务

### 为什么先做

只有统一待办，系统才能真正减少人工判断成本。

### 完成标准

- 上一步单据完成后，系统自动生成下一步待办
- 扫码交接中心不再只依赖前端推导候选项
- 仪表盘和菜单数量都来自统一待办源

## 阶段 5：打通库存与追踪

### 目标

让系统不仅能看流程，还能正确落账和追踪库存归属。

### 开发项

1. 定义库存变更节点
2. 建立在途、在库、冻结、可出货库存台账
3. 把封包、出货、交接与库存联动
4. 把异常冻结与库存冻结联动

### 为什么先做

如果没有库存台账，流程看起来完整，但管理价值仍然不够。

### 完成标准

- 关键节点能正确变更库存归属
- 异常可冻结库存
- 出货签收后库存自动扣减和归档

## 阶段 6：接企业微信与后端

### 目标

在业务对象和状态机稳定后，再接企业微信和后端持久化。

### 开发项

1. 企业微信身份绑定
2. 企业微信扫码确认回调
3. 后端接口与持久化
4. 审计日志与消息中心

### 为什么放最后

业务结构没定稳之前接接口，会把临时模型固化，后面更难改。

### 完成标准

- 交接确认可真实回写企业微信结果
- 数据不再只保存在前端本地
- 关键操作有审计日志和通知机制

## 5. 当前最优先任务清单

按优先级排序，建议下一轮先做以下 8 项：

1. 确认旧 `订单管理` 的最终定位
2. 新建 `采购批次创建` 页面
3. 新建 `客户订单创建` 页面
4. 新建 `收货管理` 一级菜单
5. 新建 `香港收货` 页面
6. 新建 `深圳收货` 页面
7. 设计统一状态机映射表
8. 设计待办自动生成规则

## 6. 当前各环节还需补充的内容

## 6.1 采购环节

- 采购批次创建入口
- 供应商正式主数据
- 金额、币种、附件、审批信息
- 发运前校验

## 6.2 在途环节

- 发运动作入口
- 到港提醒
- 节点时间轴
- 运输单据附件

## 6.3 收货环节

- 香港收货
- 深圳收货
- 实收差异复核
- 差异异常自动挂单

## 6.4 分选环节

- 分选领料前的库存校验
- 分选结果明细结构
- 损耗规则
- 分选交回审批

## 6.5 细筛环节

- 细筛规则模板
- 复检机制
- 等级调整记录
- 与封包前放行联动

## 6.6 封包环节

- 封签码规则
- 封包明细与客户要求映射
- 封包复核放行
- 待出货库存冻结

## 6.7 出货环节

- 客户订单来源
- 出货复核放行
- 物流与回单管理
- 客户签收后的归档闭环

## 6.8 扫码交接环节

- 统一待办源
- 企业微信实名强校验
- 超时规则
- 异常重试机制

## 6.9 异常环节

- 异常类型标准化
- 责任人分派规则
- 关闭说明必填
- 关闭后的复盘或归档规则

## 7. 推荐的交付顺序

推荐采用以下顺序推进，不建议打乱：

1. 先确认业务蓝图
2. 再补采购批次和客户订单入口
3. 再补收货管理
4. 再统一状态机
5. 再做自动派单和待办
6. 再补库存与追踪
7. 最后接企业微信和后端

## 8. 每阶段验收口径

### 阶段 1 验收

- 系统内可独立创建采购批次和客户订单

### 阶段 2 验收

- 在途到港后可创建正式收货记录

### 阶段 3 验收

- 任一单据推进都有明确前置规则

### 阶段 4 验收

- 上一步完成后系统可自动生成下一待办

### 阶段 5 验收

- 库存与业务单据变化一致

### 阶段 6 验收

- 企业微信确认与后端数据可真实落库

## 9. 目前不建议立即做的内容

- 不建议继续扩写旧 `Order` 演示逻辑
- 不建议先接真实企业微信接口
- 不建议先做复杂报表
- 不建议先做大量美化性页面改造

原因很简单：业务起点和流程中枢还没冻结，现在做这些会增加返工。

## 10. 下一步建议

下一步建议直接进入“阶段 1”前的细化设计，先输出以下 3 份更细的规格：

1. `采购批次 + 客户订单` 字段定义表
2. `收货管理` 页面与单据规格
3. `状态机 + 自动派单` 规则表

这 3 份规格一旦确认，就可以正式进入页面与数据结构重构阶段。
