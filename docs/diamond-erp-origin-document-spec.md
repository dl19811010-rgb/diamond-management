# 钻石 ERP 源头单据规格

## 1. 文档目的

本规格用于冻结系统最上游的两类源头单据：

- `采购批次`
- `客户订单`

本规格只处理“业务从哪里开始创建”这个问题，不展开后续收货、分选、细筛、封包、出货的详细字段。

本规格要解决的核心问题：

1. 哪些角色可以创建源头单据
2. 创建时必须录入哪些信息
3. 哪些字段由系统自动生成
4. 源头单据有哪些合法状态
5. 源头单据与后续过程单据如何关联

## 2. 源头单据总体规则

### 2.1 源头单据定义

系统仅保留两类源头单据：

- `采购批次`：供应链主线起点
- `客户订单`：销售与出货主线起点

除这两类单据外，后续单据原则上不作为源头创建入口，而应由流程推进自动生成或从待办中发起。

### 2.2 创建权限建议

| 单据 | 可创建角色 | 可修改角色 | 可确认角色 |
| --- | --- | --- | --- |
| 采购批次 | 采购员、采购主管 | 采购员、采购主管 | 采购主管 |
| 客户订单 | 销售、客户经理、跟单 | 销售、客户经理、跟单 | 销售主管、跟单主管 |

### 2.3 创建页面建议

建议新增以下正式入口：

- `新建采购批次`
- `采购批次列表`
- `新建客户订单`
- `客户订单列表`

当前旧 `订单管理` 页面不建议继续作为供应链主线入口，后续应转型为 `客户订单管理` 或拆分。

## 3. 采购批次规格

## 3.1 单据定位

采购批次是整条内部供应链主线的源头。

它回答的是：

- 这批货是从哪个供应商来的
- 属于什么石种与类别
- 预计多少重量、多少粒数
- 计划从哪里发出、何时到港
- 后面所有在途、收货、分选、细筛、封包记录归属于哪一个源头

## 3.2 建单时机

建议在以下条件满足后创建采购批次：

- 已与印度供应商确认采购
- 已明确石种类别与大致数量
- 已确认首个发运计划

## 3.3 编号规则建议

建议编号格式：

`PB + 年月日 + 三位流水`

示例：

`PB20260509001`

## 3.5 采购批次状态定义

| 状态值 | 中文含义 | 说明 |
| --- | --- | --- |
| draft | 草稿 | 已创建但未正式确认 |
| confirmed | 已确认 | 已确认采购，等待发运 |
| in_transit_to_hk | 发往香港 | 已进入印度到香港在途 |
| arrived_hk | 已到香港 | 等待香港收货或已完成香港收货 |
| transferred_to_sz | 发往深圳 | 已进入香港到深圳在途 |
| arrived_sz | 已到深圳 | 等待深圳收货入库或已收货 |
| in_processing | 生产处理中 | 已进入分选/细筛/封包主线 |
| completed | 已完成 | 已完成该批次全部出货与关闭 |
| voided | 已作废 | 作废关闭，不再推进 |
| exception | 异常冻结 | 当前批次因异常暂停推进 |

## 3.6 建单校验规则

创建采购批次时，建议至少校验以下规则：

1. 供应商不能为空
2. 石种类别不能为空
3. 预计重量必须大于 0
4. 预计粒数必须大于 0
5. 采购确认日期不能为空
6. 起运地点与首站目的地不能为空
7. 采购负责人不能为空

## 3.7 后续关联关系

采购批次创建后，后续应可关联生成：

- 在途流转单
- 香港收货单
- 深圳收货单
- 分选单
- 细筛单
- 封包单
- 交接单
- 异常单

采购批次是供应链内部主线的唯一追溯源头之一。

## 3.8 页面结构建议

### 新建采购批次页面

建议分为 4 个区块：

1. 基础信息
2. 数量与品质
3. 发运计划
4. 附件与备注

### 采购批次列表页

建议至少支持以下能力：

- 按状态筛选
- 按供应商筛选
- 按石种类别筛选
- 按日期范围筛选
- 查看后续在途与收货关联记录

## 4. 客户订单规格

## 4.1 单据定位

客户订单是销售与出货主线的源头。

它回答的是：

- 客户是谁
- 客户要什么货
- 客户对品质、粒数、封包、签收方式有什么要求
- 出货单是基于哪一笔客户需求生成的

## 4.2 建单时机

建议在以下条件满足后创建客户订单：

- 客户明确下单
- 客户已提出品质、重量或封包要求
- 业务已确认进入备货或排货

## 4.3 编号规则建议

建议编号格式：

`CO + 年月日 + 三位流水`

示例：

`CO20260509001`

## 4.6 建单校验规则

创建客户订单时，建议至少校验以下规则：

1. 客户不能为空
2. 下单日期不能为空
3. 销售负责人不能为空
4. 需求石种类别不能为空
5. 需求重量必须大于 0
6. 优先级不能为空

如果要求客户扫码签收，还建议额外校验：

7. 签收要求不能为空
8. 客户联系人或客户签收主体不能为空

## 4.7 后续关联关系

客户订单创建后，后续应可关联生成：

- 备货匹配记录
- 封包匹配记录
- 出货单
- 出货交接单
- 客户签收交接单
- 异常单

客户订单是出货链和客户履约链的唯一追溯源头之一。

## 4.8 页面结构建议

### 新建客户订单页面

建议分为 5 个区块：

1. 客户基础信息
2. 货品需求信息
3. 封包与出货要求
4. 联系与签收要求
5. 结算与备注

### 客户订单列表页

建议至少支持以下能力：

- 按状态筛选
- 按客户筛选
- 按交期筛选
- 按优先级筛选
- 查看已匹配封包和出货进度

## 5. 两类源头单据的关系

### 5.1 不是互相替代

采购批次和客户订单不是同一类对象：

- 采购批次回答“货从哪里来”
- 客户订单回答“货要发给谁”

### 5.2 在封包与出货阶段发生关联

建议在以下时点发生关联：

- 封包完成后进入可匹配库存
- 销售根据客户订单需求匹配封包
- 匹配完成后生成出货单

### 5.3 允许一个对多个关系

建议支持：

- 一个采购批次拆出多个封包
- 多个封包匹配到一个客户订单
- 一个客户订单分多次出货

## 6. 系统生成字段与人工录入字段边界

### 6.1 系统生成字段

建议以下字段统一由系统生成：

- id
- batchNo
- orderNo
- createdBy
- createdAt
- confirmedBy
- confirmedAt
- status 变更时间

### 6.2 人工录入字段

建议以下字段由业务角色录入：

- 供应商
- 石种类别
- 重量、粒数
- 客户
- 品质要求
- 出货要求
- 签收要求
- 附件与备注

### 6.3 自动带出字段

建议以下字段由主数据或当前账户自动带出：

- supplierName
- customerName
- buyerName
- salesName

## 7. 当前原型改造建议

结合当前项目现状，建议按以下方式改造：

1. 新增 `采购批次创建` 页面与 store 数据结构
2. 新增 `客户订单创建` 页面与 store 数据结构
3. 将当前 `订单管理` 重命名并改造成 `客户订单管理`
4. 后续在 `出货管理` 中不再直接维护“客户需求描述”，而是引用客户订单字段
5. 后续在 `采购管理` 中不再只是展示 mock 卡片，而是承接正式建单入口与状态推进

## 8. 本规格冻结后可直接进入的开发动作

本规格一旦确认，可以直接进入以下开发：

1. 扩展类型定义：
   - `PurchaseBatch`
   - `CustomerOrder`
2. 扩展 store：
   - 新增创建、更新、确认方法
3. 新增页面：
   - `PurchaseBatchCreate`
   - `CustomerOrderCreate`
4. 改造页面：
   - `PurchaseManagement`
   - `OrderManagement`
5. 衔接后续规格：
   - 收货管理规格
   - 状态机与自动派单规则

## 9. 待后续集中确认的业务点

本规格写完后，建议后面集中确认以下事项：

1. 客户订单是否允许不填粒数，只填重量与品质范围
2. 一个客户订单是否允许跨多个采购批次匹配
3. 是否需要在采购批次阶段就录入采购金额和币种
4. 客户签收是否必须企业微信实名确认
5. 收货差异达到什么阈值时自动挂异常

这些不影响本轮先冻结“源头单据”框架，但会影响下一阶段的规则细化。
